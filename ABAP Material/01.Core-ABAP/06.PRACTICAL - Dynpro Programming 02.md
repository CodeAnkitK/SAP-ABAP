# ABAP Dynpro (Extended)

## Detailed Creation Guide with Step-by-Step Build Process

**3 Screens, Subscreen, Messages, Custom Table, Navigation, and Table Handling**

Classic Dynpro is not just old SAP technology. It is still one of the cleanest ways to teach how SAP dialog applications actually work. A beginner who understands Dynpro properly will understand:

* screen flow
* user interaction
* PBO/PAI logic
* database updates
* navigation
* modular programming

This training demo is good because it is small enough to build in class, but rich enough to cover the real backbone of SAP screen programming. 

---

# 1) Training Goal

The idea is to build **one complete module pool application** where the learner can:

1. Start from a home screen.
2. Move to an employee maintenance screen.
3. View a demo screen for messages and subscreen behavior.
4. Read, save, update, and delete records in a custom table.
5. Understand how SAP screen logic is split into:

   * Screen layout
   * Flow logic
   * ABAP modules
   * Database operations

This is the right teaching model because beginners usually fail when they see Dynpro in fragments. They need one connected example.

---

# 2) Final Objects to Be Created

Before touching code, make the learner understand that Dynpro development is never only “writing ABAP.” It always involves multiple repository objects.

## Repository Objects

Create these objects:

* **Module Pool Program:** `ZMP_TRAINING_DEMO`
* **Transaction Code:** `ZTRN_DYNPRO_DEMO`
* **Transparent Table:** `ZTRN_EMP`
* **Message Class:** `ZMC_TRAINING`
* **Screens:**

  * `0100` – Main Navigation
  * `0150` – Subscreen
  * `0200` – Employee Maintenance
  * `0300` – Message Demo
* **GUI Statuses:**

  * `STAT100`
  * `STAT200`
  * `STAT300`
* **Title Bars:**

  * `TIT100`
  * `TIT200`
  * `TIT300`

A beginner must understand this clearly:

> **SE11** is for tables,
> **SE91** is for messages,
> **SE80/SE38** is for program logic,
> **SE51** is for screen painter,
> **SE41** is for GUI status,
> **SE93** is for transaction code.

That is the actual working model.

# 3) Step 1 – Create the Custom Table in SE11

This should be created first because screen fields and program logic will depend on it.

## Open SE11

Choose **Database Table** and create:

`ZTRN_EMP`

## Fields to Create

| Field Name  | Key | Data Type    | Length | Purpose         |
| ----------- | --: | ------------ | -----: | --------------- |
| MANDT       | Yes | CLNT         |      3 | Client          |
| EMP_ID      | Yes | NUMC         |      6 | Employee ID     |
| EMP_NAME    |  No | CHAR         |     40 | Employee Name   |
| DEPT        |  No | CHAR         |     20 | Department      |
| DESIGNATION |  No | CHAR         |     30 | Designation     |
| SALARY      |  No | CURR         |   13,2 | Salary          |
| CURRENCY    |  No | CUKY / WAERS |      5 | Currency        |
| JOIN_DATE   |  No | DATS         |      8 | Joining Date    |
| STATUS      |  No | CHAR         |      1 | Active/Inactive |

## Important Teaching Point

Do not just tell beginners to “create a table.” Explain why each field exists.

* `EMP_ID` = unique identifier
* `EMP_NAME` = display field
* `DEPT` and `DESIGNATION` = business details
* `SALARY` and `CURRENCY` = financial values
* `JOIN_DATE` = date handling
* `STATUS` = simple business flag

## Delivery and Maintenance Class

Set these properly:

* **Delivery Class:** `A` (Application table)
* **Data Browser/Table View Maintenance:** Allow display/maintenance for demo if needed
* **Enhancement Category:** Can be “Can be enhanced” or standard allowed option

## Technical Settings

Go to **Technical Settings** and maintain:

* **Data Class:** `APPL0`
* **Size Category:** `0` or `1`
* **Buffering:** No buffering needed for demo

## Activate

Save and activate the table.

## What to Explain in Class

Tell the learner:

* Transparent tables are physical database tables.
* In dialog programming, a custom table is often used to simulate business transactions.
* Without a table, screen programming becomes too theoretical.

# 4) Step 2 – Create Message Class in SE91

## Open SE91

Create message class:

`ZMC_TRAINING`

## Suggested Messages

| No. | Text                                           |
| --: | ---------------------------------------------- |
| 001 | Employee ID is required                        |
| 002 | Employee Name is required                      |
| 003 | Record saved successfully for Employee ID &1   |
| 004 | Record updated successfully for Employee ID &1 |
| 005 | Record deleted successfully for Employee ID &1 |
| 006 | Employee ID &1 not found                       |
| 007 | Screen &1 opened                               |
| 008 | Please enter valid salary                      |
| 009 | Popup subscreen displayed                      |
| 010 | Action cancelled by user                       |
| 011 | Navigation to screen &1 completed              |
| 012 | Confirm delete for employee &1                 |

## Why This Matters

This is not cosmetic. Hardcoding message texts in ABAP is sloppy design. Message classes centralize user communication.

## Classroom Point

Explain message types:

* `S` = success
* `I` = information
* `W` = warning
* `E` = error
* `A` = abort
* `X` = exit / dump-like termination

Most beginners memorize syntax but do not understand behavior. They must see what happens on screen for each message type.

# 5) Step 3 – Create Module Pool Program in SE80

## Open SE80

Create Program:

`ZMP_TRAINING_DEMO`

Choose **Module Pool**.

## Why Module Pool?

Because this is a screen-based application. Report programs are event-driven reports. Module pool programs are dialog applications.

That distinction is fundamental. If a learner misses this, they will remain confused forever.

## Initial Global Declarations

At program level, define:

* `TABLES ztrn_emp`
* OK code variables
* work area and internal table
* subscreen variable
* popup toggle
* table control declaration

## What to Explain

At the top include the theory:

* Global data is shared across screens
* `ok_code` captures user commands
* `save_ok` is used to safely process and then clear commands
* `gs_emp` holds one employee
* `gt_emp` holds many employees
* screen fields can bind directly with table work area fields

This direct binding is one of the strongest teaching advantages of Dynpro.

* `SCR3` → Demo screen

* pushing a button sends a function code
* the function code goes into `OK_CODE`
* PAI handles it
* then navigation happens

That is the dialog cycle.

# 7) Step 5 – Create Screen 0150 as Subscreen

## Purpose

This screen teaches **subscreen embedding**.

Subscreens are used when:

* screen sections are reused
* extra information is displayed inside a main screen
* modular screen design is required

## Create Screen 0150

In attributes, mark it as **Subscreen**.

## Layout

Place a few display or input fields such as:

* Employee ID
* Employee Name
* Department
* A short text like: “This is a training subscreen”

Keep it simple. The goal is structural understanding, not fancy design.

## What to Explain

A subscreen:

* does not have its own full screen flow like a standalone window
* is included inside another screen area
* still has PBO/PAI modules
* is controlled by the parent screen

This is where learners usually get confused. Be explicit.

# 11) Step 9 – Add Screen Flow Logic

This is the bridge between screen painter and ABAP logic.

## Screen 0100 Flow Logic

```abap
PROCESS BEFORE OUTPUT.
  MODULE status_0100.
  IF gv_popup_on = abap_true.
    CALL SUBSCREEN sub_pop INCLUDING sy-repid '0150'.
  ENDIF.

PROCESS AFTER INPUT.
  IF gv_popup_on = abap_true.
    CALL SUBSCREEN sub_pop.
  ENDIF.
  MODULE user_command_0100.
```

## What to Explain

* **PBO** = before screen is shown
* **PAI** = after user action
* `CALL SUBSCREEN ... INCLUDING ...` loads subscreen in PBO
* `CALL SUBSCREEN ...` in PAI transfers user actions

This is not optional. If a subscreen exists, both sides matter.

## Screen 0200 Flow Logic

```abap
PROCESS BEFORE OUTPUT.
  MODULE status_0200.

PROCESS AFTER INPUT.
  MODULE user_command_0200.
```

Simple and clean. Good for beginners.

## Screen 0300 Flow Logic

```abap
PROCESS BEFORE OUTPUT.
  MODULE status_0300.
  CALL SUBSCREEN sub_demo INCLUDING sy-repid gv_subscreen.

PROCESS AFTER INPUT.
  CALL SUBSCREEN sub_demo.
  MODULE user_command_0300.
```

## Screen 0150 Flow Logic

```abap
PROCESS BEFORE OUTPUT.
  MODULE status_0150.

PROCESS AFTER INPUT.
  MODULE user_command_0150.
```

# 12) Step 10 – Build ABAP Modules Properly

Now explain each major module in business terms.

---

## Module `status_0100 OUTPUT`

Purpose:

* Set toolbar
* Set title
* show opening message

Typical logic:

* `SET PF-STATUS 'STAT100'`
* `SET TITLEBAR 'TIT100'`

### Teaching Point

PBO is not for saving data. It is for preparing the screen.

---

## Module `user_command_0100 INPUT`

Purpose:

* process button clicks
* decide where to go next
* toggle popup/subscreen

### Important Pattern

```abap
save_ok = ok_code.
CLEAR ok_code.
```

This is standard and important.

### Why?

If you do not clear `OK_CODE`, the same command may repeat on the next cycle.

That is a classic beginner mistake.

---

## Module `status_0200 OUTPUT`

Purpose:

* prepare employee screen
* load table control data

Call:

* `SET PF-STATUS`
* `SET TITLEBAR`
* `PERFORM get_data_0200`

### Important Explanation

Refreshing internal table in PBO makes the screen display current data.

---

## Module `user_command_0200 INPUT`

Purpose:

* get record
* save record
* delete record
* clear form
* navigate

This is where business transaction logic is usually handled.

---

## Module `status_0300 OUTPUT`

Purpose:

* prepare demo screen
* ensure subscreen number exists

Why check `gv_subscreen`?
Because a subscreen call needs a valid screen number. If blank, runtime will fail.

---

## Module `user_command_0300 INPUT`

Purpose:

* show message types
* navigate
* load subscreen

This is excellent for explaining message impact.

---

# 13) Step 11 – Create Forms / Processing Routines

Now move to business logic.

---

## FORM `get_data_0200`

Purpose:

* load records into internal table for table control

Typical logic:

```abap
SELECT *
  FROM ztrn_emp
  INTO TABLE gt_emp
  UP TO 50 ROWS.
```

### Why Good for Training

It is simple enough to understand.
Do not over-engineer with complex joins for beginners.

---

## FORM `get_single_employee`

Purpose:

* fetch one employee based on entered EMP_ID

### Sequence

1. Check if EMP_ID is entered
2. `SELECT SINGLE`
3. If found, move to `gs_emp`
4. Show success/info
5. If not found, show error

### What to Teach

* validation before database call
* `sy-subrc` check
* work area population
* screen fields automatically reflect work area values

That last part is very important in Dynpro.

---

## FORM `validate_employee`

Purpose:

* ensure mandatory data is correct before save

### Suggested validations

* Employee ID cannot be blank
* Employee Name cannot be blank
* Salary must be non-negative
* default currency = INR
* default status = A

### Why This Matters

A dialog application without validation is junk.
Learners must be trained to validate before DB operations.

---

## FORM `save_employee`

Purpose:

* insert new employee or update existing employee

### Step-by-step flow

1. Validate input
2. Fill `MANDT`
3. Check if record exists
4. If exists → `UPDATE`
5. Else → `INSERT`
6. Show success message
7. Refresh table control data

### Key Point

This routine teaches difference between:

* create
* update
* refresh

Do not skip the post-save refresh. Otherwise the learner sees stale data and gets confused.

---

## FORM `delete_employee`

Purpose:

* delete record safely

### Step-by-step flow

1. Validate employee ID
2. Show popup confirmation
3. If user confirms:

   * delete from DB
   * clear work area
   * refresh table display
4. Else:

   * show cancellation message

### Why It Matters

This teaches safe delete behavior and user confirmation.

That is real-world design, not toy logic.

Here is the correct order. This is the sequence you should teach in class.

1. Create transparent table in SE11
2. Activate technical settings
3. Create message class in SE91
4. Create module pool in SE80
5. Add global data declarations
6. Create screen 0100
7. Create subscreen 0150
8. Create screen 0200
9. Create screen 0300
10. Create GUI statuses in SE41
11. Create title bars
12. Write flow logic for each screen

14. Write FORMs
15. Activate program and screens
16. Create transaction code in SE93
17. Test navigation
18. Test save/update/delete
19. Test subscreen behavior

This order prevents dependency problems.

A beginner must not just “run and see.” Testing should be methodical.

* Are buttons visible?
* Does popup toggle work?

## Screen 0200

* records inserted

# 17) Step 15 – Common Beginner Mistakes

This part matters. Training is weak if you only show the happy path.

## Mistake 1: Not clearing `OK_CODE`

Result: repeated command execution

## Mistake 2: Forgetting to activate screen or GUI status

Result: runtime issues or missing toolbar actions

## Mistake 3: Missing PAI `CALL SUBSCREEN`

Result: subscreen input not processed

## Mistake 4: No `sy-subrc` check

Result: wrong assumption that database read/write worked

## Mistake 5: No refresh after save/delete

Result: screen shows old data

## Mistake 6: Wrong screen field names

Result: binding problems between screen and ABAP fields

## Mistake 7: Treating module pool like a report

Result: conceptual confusion

A proper trainer should call these out explicitly.

# 18) Step 16 – How to Explain PBO and PAI Clearly to Beginners

Most people explain this badly. Here is the clean version:

## PBO – Process Before Output

Runs **before the screen is displayed**.

Use it for:

* setting PF-STATUS
* setting title
* filling fields/internal tables
* preparing screen display

## PAI – Process After Input

Runs **after user action** like Enter or button click.

Use it for:

* reading user commands
* validating data
* saving/updating/deleting
* navigation

### Easy Memory Trick

* **PBO = prepare**
* **PAI = process**

That usually works better than textbook definitions.
