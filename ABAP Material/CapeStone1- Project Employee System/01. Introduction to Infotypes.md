# SAP HR Infotypes: Introduction for Employee Management Projects

**Infotypes** are SAP's structured data containers specifically designed for Human Resources (HR) module. They organize employee data into logical business units, making it easy to manage complex HR information like personal data, organizational assignment, payroll results, and time records.

For your **employee system learning project**, infotypes provide the perfect foundation because they work seamlessly with **Dynpro screens**, **Menu Painter**, **BDC**, **BAPI**, and **Message classes**.

## What Are Infotypes?

An **infotype** is a **four-character code** that represents a specific type of HR data. Each infotype has:
- Predefined **data structure** (fields)
- **Time constraint** (how records are stored over time)
- **Screen control** (Dynpro screens for data entry)
- **Logical grouping** of related fields

**Examples:**
```
0000 - Actions (hiring, promotion, termination)
0001 - Organizational Assignment
0002 - Personal Data  
0006 - Addresses
0008 - Basic Pay
0009 - Bank Details
0014 - Recurring Payments/Deductions
```

## Key Infotype Concepts

### 1. Time Constraints
Infotypes use **time constraints** to manage record validity:

| Class | Behavior |
|---|---|
| **A** | One record must always exist (0001 Org Assignment) |
| **B** | One record can exist at most (0002 Personal Data) |
| **C** | Multiple records can coexist (0006 Addresses) |
| **D** | No time constraint (infotype 0310 for additional data) |

### 2. Subtypes
Many infotypes have **subtypes** for specialization:
```
0006 Addresses: 1=Home, 2=Work, 3=Emergency
0014 Payments: 1=Bonus, 2=Overtime
```

### 3. Infotype Records
Each record contains:
```
PERNR  - Personnel Number
INFTY  - Infotype Number  
SUBTY  - Subtype
OBJPS  - Object Specification
SPRPS  - Lock Indicator
ENDDA  - End Date (validity)
BEGDA  - Begin Date (validity)
SEQNR  - Sequence Number
```

## Practical Implementation Guide

### 1. **Custom Table for Employee Master**
```sql
ZEMP_MASTER
PERNR  NUMC8   - Personnel Number
NAME   CHAR40  - Employee Name
BUKRS  CHAR4   - Company Code
```

### 2. **Dynpro Screens Using PA30 Style**
```
Screen 100 - Employee Selection
Screen 200 - Personal Data (0002)
Screen 300 - Basic Pay (0008)
Screen 400 - Address (0006)
```

### 3. **Menu Painter Structure**
```
Main Menu
├── Employee → Create (F5)
├── Employee → Change (F6) 
├── Employee → Display (F7)
├── Utils → Messages (F8)
└── Exit (Shift+F4)
```

### 4. **BDC for PA40/PA30 Transactions**
```abap
" Record PA40 Create Employee
PERFORM bdc_dynpro USING 'MP000800' '2000'.
PERFORM bdc_field  USING 'P0008-BET01' '50000'.
CALL TRANSACTION 'PA40' USING lt_bdcdata MODE 'N'.
```

### 5. **BAPI for HR Operations**
```abap
" BAPI_EMPLOYEE_ENQUEUE
" BAPI_EMPLOYEE_CHANGE
" BAPI_HRMASTER_SAVE
CALL FUNCTION 'BAPI_EMPLOYEE_GETDATA'
  EXPORTING
    employeenumber = lv_pernr
  TABLES
    employee_data  = lt_data
    return         = lt_return.
```

### 6. **Message Class ZEMP_MSG**
```
001 Employee &1 created successfully
002 Invalid personnel number &1
003 Salary must be greater than 10000
004 Address subtype &1 required
```

## Sample Code Skeleton

```abap
PROGRAM zmp_employee_sys MESSAGE-ID zemp_msg.

" Global data
DATA: gv_pernr TYPE pernr_d,
      gs_p0002 TYPE p0002,
      gs_p0008 TYPE p0008.

MODULE status_0100 OUTPUT.
  SET PF-STATUS 'MAIN_MENU'.
  SET TITLEBAR 'TITLE_EMP'.
ENDMODULE.

MODULE create_employee INPUT.
  " Validate input
  IF gv_pernr IS INITIAL.
    MESSAGE e002.
  ENDIF.
  
  " Call PA40 BDC or BAPI
  PERFORM process_pa40 USING gv_pernr.
  
  MESSAGE s001 WITH gv_pernr.
ENDMODULE.
```
